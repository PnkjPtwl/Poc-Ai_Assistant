# AI Sales Engagement Platform — Full Stack POC

## PROJECT OVERVIEW
Build a fully functional AI-powered B2B Sales Engagement Platform. This is a unified workspace for sales reps to manage outreach, automate email cadences, get AI coaching during sales plays, and track pipeline forecasts. The platform uses real AI (Groq), real email sending (SMTP), and real persistent data (Supabase).

---

## TECH STACK

### Frontend
- Vite + React + TypeScript
- Tailwind CSS (no component library, pure utility classes)
- Zustand (global state)
- React Router v6 (client-side routing)
- react-hot-toast (notifications)
- date-fns (date manipulation)
- lucide-react (icons)
- axios (HTTP client)

### Backend
- FastAPI (Python)
- aiosmtplib (async SMTP for email sending from user's own account)
- apscheduler (scheduled email jobs)
- groq (Python SDK for Groq AI)
- supabase-py (Supabase client)
- python-dotenv

### Infrastructure
- Supabase (PostgreSQL + realtime)
- Groq API: model = `llama-3.3-70b-versatile`

## DATABASE SCHEMA (Supabase/PostgreSQL)

Create all tables with RLS disabled for POC. Use UUIDs as PKs. Run this SQL in Supabase SQL editor:

```sql
-- Reps (sales users)
CREATE TABLE reps (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  email TEXT NOT NULL UNIQUE,
  role TEXT DEFAULT 'Account Executive',
  quota NUMERIC DEFAULT 500000,
  avatar_initials TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Companies
CREATE TABLE companies (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  industry TEXT,
  website TEXT,
  employee_count INT,
  location TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Contacts
CREATE TABLE contacts (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  email TEXT NOT NULL,
  phone TEXT,
  whatsapp_number TEXT,
  company_id UUID REFERENCES companies(id),
  title TEXT,
  preferred_language TEXT DEFAULT 'English',
  timezone TEXT DEFAULT 'America/New_York',
  activity_score INT DEFAULT 0,
  last_contacted_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Deals (CRM)
CREATE TABLE deals (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  title TEXT NOT NULL,
  contact_id UUID REFERENCES contacts(id),
  company_id UUID REFERENCES companies(id),
  rep_id UUID REFERENCES reps(id),
  stage TEXT CHECK (stage IN ('Prospecting','Qualification','Discovery','Proposal','Negotiation','Closed Won','Closed Lost')),
  amount NUMERIC DEFAULT 0,
  probability INT DEFAULT 0,
  close_date DATE,
  forecast_category TEXT CHECK (forecast_category IN ('Commit','Best Case','Pipeline','Omit')) DEFAULT 'Pipeline',
  next_step TEXT,
  notes TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- Email templates
CREATE TABLE email_templates (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  category TEXT,
  subject_template TEXT NOT NULL,
  body_template TEXT NOT NULL,
  variables JSONB DEFAULT '[]',
  language TEXT DEFAULT 'English',
  open_rate NUMERIC DEFAULT 0,
  reply_rate NUMERIC DEFAULT 0,
  times_used INT DEFAULT 0,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Emails
CREATE TABLE emails (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  deal_id UUID REFERENCES deals(id),
  contact_id UUID REFERENCES contacts(id),
  rep_id UUID REFERENCES reps(id),
  thread_id TEXT,
  subject TEXT NOT NULL,
  body TEXT NOT NULL,
  status TEXT CHECK (status IN ('draft','scheduled','sent','opened','replied','bounced')) DEFAULT 'draft',
  scheduled_at TIMESTAMPTZ,
  sent_at TIMESTAMPTZ,
  opened_at TIMESTAMPTZ,
  replied_at TIMESTAMPTZ,
  language TEXT DEFAULT 'English',
  ai_generated BOOLEAN DEFAULT false,
  template_id UUID REFERENCES email_templates(id),
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Call transcripts (demo data)
CREATE TABLE call_transcripts (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  contact_id UUID REFERENCES contacts(id),
  deal_id UUID REFERENCES deals(id),
  rep_id UUID REFERENCES reps(id),
  duration_seconds INT,
  transcript TEXT NOT NULL,
  summary TEXT,
  language_detected TEXT DEFAULT 'English',
  sentiment TEXT CHECK (sentiment IN ('Positive','Neutral','Negative')),
  key_topics JSONB DEFAULT '[]',
  call_date TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Sales plays
CREATE TABLE plays (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  type TEXT CHECK (type IN ('Cold Outreach','Cross-Sell','Upsell','Onboarding','Renewal','Re-Engagement')),
  description TEXT,
  target_stages TEXT[],
  total_steps INT DEFAULT 0,
  duration_days INT DEFAULT 30,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Flows (automated multi-step sequences)
CREATE TABLE flows (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  play_id UUID REFERENCES plays(id),
  name TEXT NOT NULL,
  description TEXT,
  trigger_type TEXT CHECK (trigger_type IN ('deal_stage_change','time_elapsed','email_opened','email_replied','manual')),
  trigger_condition JSONB DEFAULT '{}',
  status TEXT CHECK (status IN ('active','paused','draft')) DEFAULT 'active',
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Flow steps
CREATE TABLE flow_steps (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  flow_id UUID REFERENCES flows(id) ON DELETE CASCADE,
  step_number INT NOT NULL,
  channel TEXT CHECK (channel IN ('email','whatsapp','call','task')) DEFAULT 'email',
  delay_days INT DEFAULT 0,
  subject_template TEXT,
  body_template TEXT,
  action_description TEXT,
  template_id UUID REFERENCES email_templates(id),
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Flow enrollments
CREATE TABLE flow_enrollments (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  flow_id UUID REFERENCES flows(id),
  contact_id UUID REFERENCES contacts(id),
  deal_id UUID REFERENCES deals(id),
  rep_id UUID REFERENCES reps(id),
  current_step INT DEFAULT 1,
  status TEXT CHECK (status IN ('active','completed','paused','exited')) DEFAULT 'active',
  enrolled_at TIMESTAMPTZ DEFAULT NOW(),
  next_action_at TIMESTAMPTZ,
  completed_at TIMESTAMPTZ
);

-- Tasks (consolidated daily to-do queue)
CREATE TABLE tasks (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  rep_id UUID REFERENCES reps(id),
  contact_id UUID REFERENCES contacts(id),
  deal_id UUID REFERENCES deals(id),
  flow_enrollment_id UUID REFERENCES flow_enrollments(id),
  type TEXT CHECK (type IN ('email','call','whatsapp','review','custom')) NOT NULL,
  title TEXT NOT NULL,
  description TEXT,
  due_date DATE DEFAULT CURRENT_DATE,
  priority TEXT CHECK (priority IN ('urgent','high','normal','low')) DEFAULT 'normal',
  status TEXT CHECK (status IN ('pending','snoozed','completed','dismissed')) DEFAULT 'pending',
  snoozed_until DATE,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  completed_at TIMESTAMPTZ
);

-- Play guidelines (AI-generated coaching content)
CREATE TABLE play_guidelines (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  play_id UUID REFERENCES plays(id),
  deal_stage TEXT,
  step_number INT,
  channel TEXT,
  guideline_title TEXT,
  guideline_content TEXT,
  ai_generated BOOLEAN DEFAULT true,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Forecast entries
CREATE TABLE forecast_entries (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  rep_id UUID REFERENCES reps(id),
  deal_id UUID REFERENCES deals(id),
  period TEXT,
  year INT,
  forecast_category TEXT CHECK (forecast_category IN ('Commit','Best Case','Pipeline','Omit')),
  rep_override_amount NUMERIC,
  manager_override_amount NUMERIC,
  notes TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);
```

## CONTEXT ASSEMBLY (Critical)

In `services/groq_service.py`, before every AI call, build a full context object by fetching fresh data from Supabase. Never cache this — always fetch live.

For a given deal_id + contact_id, assemble the following and inject it into every Groq system prompt:

```python
context = {
  "contact": {
    "name": ..., "title": ..., "preferred_language": ...,
    "timezone": ..., "activity_score": ...
  },
  "company": {
    "name": ..., "industry": ..., "employee_count": ..., "location": ...
  },
  "deal": {
    "title": ..., "stage": ..., "amount": ..., "probability": ...,
    "close_date": ..., "forecast_category": ..., "next_step": ...,
    "notes": ..., "days_in_stage": ...  # computed: now - updated_at
  },
  "email_history": [
    # Last 5 emails for this deal, sorted by sent_at desc
    {
      "subject": ..., "body_snippet": ...,  # first 200 chars of body
      "status": ..., "sent_at": ..., "days_ago": ...
    }
  ],
  "call_transcripts": [
    # All transcripts for this contact, sorted by call_date desc
    # Full transcript text only for the most recent one
    # Summary only for the rest
    {
      "summary": ..., "key_topics": [...], "sentiment": ...,
      "call_date": ..., "days_ago": ...,
      "full_transcript": ...  # only on index 0
    }
  ],
  "last_contact_days_ago": ...,  # min of most recent email sent_at and most recent call_date
  "tasks_completed": [
    # Last 5 completed tasks for this deal
    { "type": ..., "title": ..., "completed_at": ... }
  ]
}
```

Format this as a structured text block injected at the top of every Groq system prompt:

```
DEAL CONTEXT:
- Deal: {title} | Stage: {stage} | Amount: ${amount} | Close: {close_date}
- Days in current stage: {days_in_stage}
- Forecast category: {forecast_category}
- Next step noted: {next_step}
- Rep notes: {notes}

CONTACT: {name}, {title} at {company_name} ({industry}, {employee_count} employees, {location})
- Preferred language: {preferred_language} | Timezone: {timezone}
- Engagement score: {activity_score}/100
- Last contacted: {last_contact_days_ago} days ago

EMAIL HISTORY (most recent first):
1. [{days_ago}d ago] "{subject}" — {status}
   Preview: "{body_snippet}"
2. ...

CALL HISTORY (most recent first):
1. [{days_ago}d ago] Sentiment: {sentiment} | Topics: {key_topics}
   Summary: {summary}
   Full transcript: {full_transcript}
2. [{days_ago}d ago] Sentiment: {sentiment}
   Summary: {summary}

COMPLETED ACTIONS:
- [{completed_at}] {type}: {title}
- ...
```

Cap total assembled context at 3000 tokens. If email history or transcript is very long, truncate body_snippet to 150 chars and use only the summary (not full transcript) for older calls. The most recent call always gets its full transcript included.

This context assembly function must be a single reusable async function `assemble_deal_context(contact_id, deal_id, db)` called at the start of every AI endpoint handler.

**Flows**
- GET /flows — list all flows with step counts and enrollment counts
- GET /flows/{id} — flow + steps detail
- POST /flows/{id}/enroll — enroll contact + deal
- GET /flows/{id}/enrollments — list enrollments
- PATCH /flow_enrollments/{id}/advance — manually advance to next step

**Deals**
- GET /deals?rep_id=&stage=&forecast_category= — with contact and company joined
- GET /deals/{id} — full deal detail with timeline of emails and tasks
- PATCH /deals/{id} — update stage, forecast_category, amount, next_step, notes, close_date

**Contacts**
- GET /contacts — with company join
- GET /contacts/{id} — with deal history, email history, task history

**Plays**
- GET /plays — with steps and guidelines
- GET /plays/{id}/guidelines — guidelines for a play, grouped by stage

**Forecast**
- GET /forecast?rep_id=&period= — all deals grouped by forecast_category with totals
- PATCH /forecast/deals/{id}/category — update forecast category
- GET /forecast/summary?period= — team rollup per rep: commit total, best case, pipeline

**Calls**
- GET /calls?contact_id=&deal_id= — list transcripts
- GET /calls/{id} — full transcript with summary
- PATCH /calls/{id}/summarize — trigger AI summarization

### SMTP Email Service
In `services/email_service.py`:
- Use `aiosmtplib` for async sending

- Handle scheduling via APScheduler (persist scheduled jobs, load on startup)

## WHATSAPP INTEGRATION (replacing LinkedIn)

For any task or action of type `whatsapp`:
1. User clicks "Generate WhatsApp Message" on a task or in the composer
2. Frontend calls POST /ai/whatsapp-message with contact_id, deal_id, intent
3. Backend assembles full deal context (same context assembly function), passes to Groq
4. Groq generates a concise conversational WhatsApp message (max 150 words, warm tone, no formal salutations)
5. Backend returns: `{message: "...", whatsapp_url: "https://wa.me/+{e164_number}?text={url_encoded_message}"}`
6. Frontend shows message preview with "Open WhatsApp" button → opens wa.me in new tab
7. User sends manually in WhatsApp Web/App
8. Frontend marks task as completed (optimistic)

WhatsApp number must be stored in E.164 format in the contacts table (+country_code + number, no spaces or dashes).

## IMPORTANT IMPLEMENTATION NOTES

1. **Active rep** — Settings page has rep selector. Store active rep ID in Zustand + localStorage. All API calls include rep_id query param or request body field.

2. **Context assembly is mandatory** — Every single AI endpoint must call `assemble_deal_context()` before building the Groq prompt. No AI call should go to Groq without full deal + contact + history context.

3. **SMTP Test Connection** — Settings "Send Test Email" button must send a real email to the rep's own email address and show success/failure toast.

4. **AI Loading States** — All Groq calls show a distinct loading state: animated pulsing "✦ AI is drafting..." indicator. Not a generic spinner.

5. **Email → Task sync** — When an email is sent (status set to 'sent'), automatically mark the linked task (if any) as 'completed'.

6. **WhatsApp E.164 format** — Seed contacts with properly formatted whatsapp_number values. wa.me links break without correct format.

7. **Forecast real-time totals** — Commit/Best Case/Pipeline totals recompute client-side instantly on category change (optimistic), confirmed after server response.

8. **APScheduler startup** — On FastAPI startup, query all emails where status='scheduled' AND scheduled_at > NOW(), register each as an APScheduler job. On execution, call the SMTP send function and update status to 'sent'.

9. **CORS** — `allow_origins=["*"]` in development.

10. **Error handling** — All API errors surface as react-hot-toast notifications. SMTP failures must include the specific error reason (auth failed, timeout, etc.).

11. **Seed idempotency** — Check existence before inserting. Print a full summary table of counts inserted per entity type.

12. **days_in_stage computation** — Compute as `(NOW() - deals.updated_at).days` in the context assembly function. Include in every deal context block passed to Groq.
